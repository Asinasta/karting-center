# TASK-01. Сгенерировать с ИИ требования к MVP (user stories + основные сценарии)

## Цель

По брифу заказчика (`01-analysis/0-customer-brief/customer-brief.md`) сгенерировать с помощью ИИ полный набор требований к MVP мобильного приложения клиента картинг-центра «Апекс»: бизнес-требования, функциональные и нефункциональные требования, user stories и основные пользовательские сценарии (use cases).

## Исходные требования

Единственный вход — бриф заказчика: 14 картов, новичковый слот ≤ 8 мест, отмена не позднее чем за 2 часа, оплата офлайн на месте, горизонт записи 7 дней, отмена заезда центром с уведомлением.

## Что сделано

Требования сгенерированы и разложены в `01-analysis/` по этапам анализа:

- `01-analysis/1-elicitation/` — вопросы заказчику, описание домена, глоссарий.
- `01-analysis/2-requirements/business-requirements.md` — бизнес-требования (BR).
- `01-analysis/2-requirements/functional-requirements.md` — функциональные требования (FR) с приоритетами Must/Should.
- `01-analysis/2-requirements/non-functional-requirements.md` — NFR (производительность, надёжность, безопасность, offline, a11y).
- `01-analysis/2-requirements/user-stories.md` — **user stories**.
- `01-analysis/2-requirements/use-cases.md` — **основные сценарии** с базовыми, альтернативными и исключительными потоками (запись на заезд, отмена, авторизация по OTP и т.д.).

После генерации проведены два раунда ИИ-аудита (роль «Системный аналитик / Product Owner»): проверка на неподтверждённые числа (помечены как «предположение»), расхождения терминов между FR / use cases / design brief / API, отсутствие в клиентском MVP функций маршала/владельца/админки, покрытие всех Must FR историями и сценариями. Найденные замечания исправлены.

## Промпты

Все отправленные промпты — в `docs/prompts/`:

- `chat-01-requirements-generation.txt` — первичная генерация требований по брифу и наполнение этапов 3–5 (design brief, data model, mobile spec).
- `chat-02-requirements-audit-and-dev-plans.txt` — проверка покрытия Must FR, чистки MVP от чужих ролей.
- `chat-03-analysis-review-and-plan-audit.txt` — аудит аналитики по чек-листу цифровой гигиены и фиксация исправлений.

## Проверка вручную

- Прочитаны user stories и use cases: каждый Must FR имеет хотя бы одну историю и сценарий.
- Все «магические числа» (тайминги push `[24, 2]`, rate-limit OTP 30–60 с, p95-бюджеты) помечены как предположения с источником.
- В MVP нет функций маршала, владельца и админки — только клиентские сценарии.
