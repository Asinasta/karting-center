## **Отчёт**

### **Что реализовано**

**Ядро** (`client/lib/core/`):

- `ApiClient` — GET/POST/PATCH/DELETE, публичные запросы без токена, защищённые с `Authorization: Bearer`; на `401` ровно один refresh и один повтор запроса.
- `AppFailure` — типизированные ошибки со всеми 13 кодами из `common/models.yaml`, `retry_after`, `double_booking.details.booking_id`, русские UI-сообщения.
- `SecureTokenStorage` (flutter_secure_storage), `LocalCache` — in-memory read-only fallback для offline stale (решение открытого вопроса №4 плана: memory-кэш для MVP).

**Фичи** (структура `data / domain / presentation` по фичам, UI не ходит в HTTP напрямую):

- `session` — `SessionRepository` (единственная точка работы с токенами, single-flight refresh, очистка storage при неуспехе), `SessionController`, splash.
- `auth` — SCR-001: телефон → код → имя для нового клиента (по `validation_error` от сервера), согласие/политика, countdown по `retry_after`, повторная отправка кода.
- `slots` — SCR-002 (сортировка по `start_at`, disabled CTA для полных/отменённых, offline stale, pull-to-refresh), BS-001 (период, тип трассы, маршал из `GET /marshals`, «только свободные», применить/сбросить), SCR-003 (все поля, перезагрузка перед бронированием, карта).
- `booking` — SCR-004 (1–3 места через `AvailabilityPolicy`, экипировка на место, превью цены, `Idempotency-Key` с переиспользованием при неопределённом результате и сбросом при смене состава), BS-002 (только `price_total` из API), SCR-005 (пагинация limit=20, группировка предстоящие/прошедшие/отменённые), SCR-006, BS-003 (правило 2 часов, финальный статус из API).
- `map` — BS-004: отрисовка `geometry` на canvas + pin точки сбора, текстовый fallback, открытие внешней карты (решение открытого вопроса №2 плана — плагин карт не согласован).
- `profile` — SCR-007: имя через `PATCH /profile` (телефон туда не отправляется), смена телефона через отдельный OTP-flow, logout и удаление аккаунта с очисткой storage только после ответа API.
- `notifications` — заглушка `PushService`: флаг «permission запрашивался», вызов после первой брони, retry `registerPushToken` при загрузке профиля. Реальный плагин push не согласован (открытый вопрос плана), поэтому системный запрос и получение токена — точка расширения.

**Роутер**: auth gate в `redirect` go_router — `/slots/:id/book`, `/bookings`*, `/profile` требуют сессию, гость уходит на `/auth?return=...` и возвращается после OTP.

### **Как проверить**

Оба процесса уже запущены мной и работают: backend на `http://localhost:8080`, клиент на `http://127.0.0.1:3000`. Если нужно перезапустить:

*# PowerShell №1*

cd C:\Users\Asinasta\Downloads\karting-center\backend

.\.venv\Scripts\python.exe manage.py run

*# PowerShell №2*

cd C:\Users\Asinasta\Downloads\karting-center\client

flutter run -d web-server --web-hostname 127.0.0.1 --web-port 3000 --dart-define=API_BASE_URL=http://localhost:8080

Сценарий smoke: откройте `http://127.0.0.1:3000` → список заездов → фильтры → карточка → «Записаться» → OTP (телефон любой в формате `+7...`, dev-код **0000**, для нового номера появится поле имени + согласие) → форма брони → успех → «Мои записи» → детали → отмена → профиль → выход. Гейты: `flutter analyze` и `flutter test` в `client/` — оба зелёные.
