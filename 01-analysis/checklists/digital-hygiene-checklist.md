# Чек-лист цифровой гигиены

Перед передачей в разработку проверить аналитику по пунктам ниже.

## Галлюцинации и достоверность

- [ ] Нет выдуманных интеграций, сервисов или SDK, которых не было в брифе, уточнениях или явных решениях команды.
- [ ] Все технические решения, добавленные командой (например, Flutter, OpenAPI, системный push), помечены как решение команды или требование реализации.
- [ ] Нет ссылок на несуществующие стандарты, нормативы или обязательства.
- [ ] Все числа имеют источник: бриф, R-решение или пометку «предположение».
- [ ] Все ключевые сущности и термины есть в глоссарии `1-elicitation/domain-description.md`.
- [ ] Термины используются единообразно: заезд, слот, конфигурация трассы, маршал, экипировка, бронь.
- [ ] OpenAPI использует `marshals`, а не `instructors`.

## Корректность требований

- [ ] Функциональные требования описывают поведение системы, а нефункциональные — качество, ограничения и эксплуатационные свойства.
- [ ] Каждое FR связано с user story, use case, API или mobile spec, по которым можно проверить реализацию.
- [ ] Каждое NFR проверяемо через критерий, ограничение, тест, бюджет производительности или архитектурное правило.
- [ ] Use cases содержат основной, альтернативные и исключительные потоки.
- [ ] Требования не противоречат друг другу и брифу заказчика.
- [ ] Публичный просмотр слотов не требует авторизации; персональные операции защищены Bearer auth.
- [ ] Смена телефона описана отдельным OTP-flow, а не прямым `PATCH phone`.
- [ ] `createBooking` использует `Idempotency-Key`.
- [ ] `Booking.slot` описан как snapshot для истории брони.
- [ ] Push об отмене центром отделён от push-напоминаний.
- [ ] Карта трассы имеет fallback без интерактивной карты.

## Полнота

- [ ] Закрыты все пробелы из `1-elicitation/customer-questions.md`.
- [ ] Явно зафиксировано, что НЕ входит в MVP.
- [ ] В MVP нет интерфейса маршала и владельца.
- [ ] Онлайн-оплата, оценки маршалов, авто-погода и лояльность вынесены в Phase 2.
- [ ] Профиль, удаление аккаунта, push, отмена центром и late cancel покрыты сценариями и API-контрактом.
- [ ] Edge cases записи покрывают overbooking, double booking, slot cancelled, slot started, timeout и offline/stale данные.
