- [x] Нет выдуманных интеграций, сервисов или SDK, которых не было в брифе, уточнениях или явных решениях команды.
- [x] Все технические решения, добавленные командой (например, Flutter, OpenAPI, системный push), помечены как решение команды или требование реализации.
- [x] Нет ссылок на несуществующие стандарты, нормативы или обязательства.
- [x] Все числа имеют источник: бриф, R-решение или пометку «предположение».
- [x] Все ключевые сущности и термины есть в глоссарии `1-elicitation/domain-description.md`.
- [x] Термины используются единообразно: заезд, слот, конфигурация трассы, маршал, экипировка, бронь.

## Корректность требований

- [x] Функциональные требования описывают поведение системы, а нефункциональные — качество, ограничения и эксплуатационные свойства.
- [x] Каждое FR связано с user story, use case, API или mobile spec, по которым можно проверить реализацию.
- [x] Каждое NFR проверяемо через критерий, ограничение, тест, бюджет производительности или архитектурное правило.
- [x] Use cases содержат основной, альтернативные и исключительные потоки.
- [x] Требования не противоречат друг другу и брифу заказчика.
- [x] Публичный просмотр слотов не требует авторизации; персональные операции защищены Bearer auth.
- [x] Смена телефона описана отдельным OTP-flow, а не прямым `PATCH phone`.
- [x] `createBooking` использует `Idempotency-Key`.
- [x] `Booking.slot` описан как snapshot для истории брони.
- [x] Push об отмене центром отделён от push-напоминаний.
- [x] Карта трассы имеет fallback без интерактивной карты.
