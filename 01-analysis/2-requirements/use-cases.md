# Use cases / сценарии

> Этап 2. Детальные сценарии взаимодействия клиента с системой. Включают альтернативные потоки и ошибки.

## UC-1: Запись на заезд

- **Актор:** Клиент
- **Предусловия:** Клиент авторизован; есть слот в статусе `scheduled`.
- **Триггер:** Клиент хочет забронировать место на заезд.

### Основной поток

1. Клиент открывает список слотов.
2. При необходимости применяет фильтры.
3. Открывает карточку заезда.
4. Выбирает число мест: 1–3 (лимит 3 — предположение).
5. Для каждого места выбирает экипировку: своя / прокатная.
6. Нажимает «Записаться».
7. Flutter-клиент отправляет `POST /bookings` с `Idempotency-Key`.
8. API атомарно проверяет места и прокатную экипировку.
9. API создаёт бронь, фиксирует `price_total`.
10. Клиент видит экран успеха и бронь в «Моих записях».

### Альтернативные потоки

- **A1. Вся экипировка своя.** `rental_count = 0`, прокатный фонд не уменьшается.
- **A2. Часть мест с прокатом.** Проверяется только нужное число прокатных комплектов.
- **A3. Фильтр не дал результатов.** Показывается empty state и действие «Сбросить фильтры».

### Исключения

- **E1. Не хватает мест.** API возвращает `409 slot_full`, клиент показывает актуальные свободные места.
- **E2. Не хватает прокатной экипировки.** API возвращает `409 slot_full` с `free_rental_gear`, клиент предлагает уменьшить прокат.
- **E3. Слот заполнился параллельно.** API отклоняет запрос, клиент обновляет слот.
- **E4. Слот отменён центром.** API возвращает `410 slot_cancelled`, CTA записи отключается.
- **E5. Таймаут.** Клиент повторяет запрос с тем же `Idempotency-Key`, дубль не создаётся.

### Постусловия

- При успехе создана активная бронь.
- При ошибке состояние слота не изменилось.

## UC-2: Отмена брони

- **Актор:** Клиент
- **Предусловия:** У клиента есть предстоящая активная бронь.
- **Триггер:** Клиент решает отменить участие.

### Основной поток

1. Клиент открывает «Мои записи».
2. Выбирает активную бронь.
3. Нажимает «Отменить».
4. Система показывает подтверждение отмены всей брони.
5. Клиент подтверждает.
6. API определяет тип отмены по серверному времени.
7. При ранней отмене бронь переходит в `cancelled`, места и прокатная экипировка возвращаются в слот.
8. Клиент видит обновлённый статус.

### Альтернативные потоки

- **A1. Поздняя отмена.** Если до старта меньше порога ранней отмены, бронь получает `late_cancel`, места не освобождаются.
- **A2. Клиентский прогноз отличается от серверного.** Применяется статус, который вернул сервер.

### Исключения

- **E1. Заезд уже стартовал.** API возвращает `slot_started`; отмена недоступна.
- **E2. Бронь уже отменена.** API возвращает `already_cancelled`; клиент показывает текущий статус.
- **E3. Нет сети.** Отмена офлайн запрещена; показывается Error/Retry.

## UC-3: Просмотр и фильтрация слотов

- **Актор:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет найти подходящий заезд.

### Основной поток

1. Клиент открывает экран «Заезды».
2. Приложение запрашивает `GET /slots` с дефолтом: ближайшие 7 дней.
3. Список отображается по возрастанию времени старта.
4. Для заполненных и отменённых центром слотов CTA записи отображается неактивной.
5. Клиент открывает фильтры.
6. Выбирает дату/период, тип трассы, маршала, «только свободные».
7. Приложение обновляет список.

### Альтернативные потоки

- **A1. Сброс фильтров.** Клиент возвращается к дефолтному списку на 7 дней.
- **A2. Только свободные.** Заполненные и отменённые слоты скрываются.
- **A3. Нет справочника маршалов.** Фильтр по маршалу скрывается.

### Исключения

- **E1. Нет слотов.** Empty state «Пока нет доступных заездов».
- **E2. Сеть недоступна.** Показывается кэш с пометкой «данные могли устареть».
- **E3. Ошибка сервера.** Показывается единый Error/Retry.

## UC-4: Регистрация и вход по OTP

- **Актор:** Клиент
- **Предусловия:** Клиент не авторизован.
- **Триггер:** Клиент запускает приложение без активной сессии.

### Основной поток

1. Клиент вводит номер телефона.
2. Flutter-клиент вызывает `POST /auth/otp`.
3. Клиент вводит SMS-код.
4. Для нового клиента приложение запрашивает имя.
5. Flutter-клиент вызывает `POST /auth/verify`.
6. API возвращает пару access/refresh token.
7. Клиент попадает на экран «Заезды».

### Альтернативные потоки

- **A1. Сессия уже есть.** При наличии валидного refresh token клиент сразу попадает в авторизованную зону.
- **A2. Access token истёк.** Клиент вызывает `POST /auth/refresh` с `refresh_token` и повторяет исходный запрос.

### Исключения

- **E1. Неверный код.** API возвращает `invalid_code`, поле кода остаётся активным.
- **E2. Rate limit.** API возвращает `rate_limit`, клиент показывает время до повторной отправки.
- **E3. Refresh token истёк.** Клиент очищает secure storage и возвращает пользователя на вход.

## UC-5: Просмотр карточки заезда и карты

- **Актор:** Клиент
- **Предусловия:** Клиент авторизован; слот доступен в списке.
- **Триггер:** Клиент хочет проверить детали перед записью.

### Основной поток

1. Клиент открывает карточку заезда.
2. Flutter-клиент вызывает `GET /slots/{slotId}`.
3. Приложение показывает дату/время, конфигурацию трассы, тип, описание, маршала, цену, места, прокатную экипировку и место сбора.
4. Клиент открывает карту трассы.
5. Приложение показывает pin места сбора и геометрию трассы.

### Альтернативные потоки

- **A1. Карта недоступна.** Клиент видит текстовый fallback: адрес/ориентир и координаты.
- **A2. Слот заполнен или отменён центром.** CTA «Записаться» неактивен.

### Исключения

- **E1. Ошибка загрузки карточки.** Показывается Error/Retry.
- **E2. Нет сети.** Для уже загруженной карточки показывается Offline stale.

## UC-6: Просмотр истории в «Моих записях»

- **Актор:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет проверить свои активные и прошлые записи.

### Основной поток

1. Клиент открывает «Мои записи».
2. Flutter-клиент вызывает `GET /bookings` с параметрами пагинации.
3. Приложение показывает предстоящие, прошедшие и отменённые брони.
4. В карточке брони отображаются статус, параметры слота, количество мест, выбор экипировки и итоговая цена.
5. Клиент открывает детали выбранной брони.

### Альтернативные потоки

- **A1. История пуста.** Показывается empty state.
- **A2. Есть следующая страница.** Клиент догружает следующую страницу без потери текущего списка.

### Исключения

- **E1. Сеть недоступна.** Показывается кэш с пометкой «данные могли устареть».
- **E2. Ошибка сервера.** Показывается Error/Retry.

## UC-7: Отмена заезда центром

- **Актор:** Клиент
- **Предусловия:** У клиента есть бронь на слот, который центр отменил в существующей инфраструктуре.
- **Триггер:** Клиент получает push или открывает историю записей.

### Основной поток

1. Клиент получает push об отмене заезда центром.
2. Клиент открывает детали брони.
3. Flutter-клиент получает бронь со статусом `cancelled_by_center` и причиной отмены.
4. Приложение показывает текст «Заезд отменён центром» и причину.
5. CTA повторной записи на этот слот недоступен.

### Альтернативные потоки

- **A1. Push запрещён.** Клиент видит статус и причину при следующем открытии «Моих записей».

### Исключения

- **E1. Клиент пытается записаться на отменённый слот.** API возвращает `slot_cancelled`, клиент показывает, что заезд отменён центром.

## UC-8: Управление профилем

- **Актор:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет изменить личные данные, выйти или удалить аккаунт.

### Основной поток

1. Клиент открывает «Профиль».
2. Flutter-клиент вызывает `GET /profile`.
3. Клиент редактирует имя или номер телефона.
4. При смене телефона клиент подтверждает новый номер по OTP.
5. Flutter-клиент вызывает `PATCH /profile`.
6. Приложение показывает обновлённые данные.

### Альтернативные потоки

- **A1. Выход из аккаунта.** Клиент нажимает «Выйти», приложение очищает secure storage и возвращает на вход.
- **A2. Удаление аккаунта.** Клиент подтверждает удаление; API отменяет активные брони, а прошедшие анонимизирует; приложение очищает secure storage.

### Исключения

- **E1. Ошибка OTP при смене телефона.** Клиент остаётся на подтверждении телефона.
- **E2. Ошибка удаления аккаунта.** Показывается Error/Retry, токены не очищаются до подтверждения удаления API.

## Матрица ошибок ключевых операций

| Код | HTTP | Операция | UX-реакция | Retry |
| :-- | :-- | :-- | :-- | :-- |
| `slot_full` | 409 | createBooking | Показать актуальные места/экипировку | Нет, изменить параметры |
| `double_booking` | 409 | createBooking | Перейти к существующей брони | Нет |
| `slot_cancelled` | 410 | createBooking | Показать, что заезд отменён центром | Нет |
| `slot_started` | 422 | create/cancel | Сообщить, что заезд уже начался | Нет |
| `already_cancelled` | 409 | cancelBooking | Показать текущий статус | Нет |
| `invalid_code` | 400/401 | verifyOtp | Оставить поле кода активным | Да |
| `rate_limit` | 429 | auth | Показать время до повтора | После `retry_after` |
| `unauthorized` | 401 | any | Refresh token или переход на вход | Да |
| `server_error` | 5xx | any | Error/Retry; для брони — тот же Idempotency-Key | Да |
